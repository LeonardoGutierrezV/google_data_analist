# Guía en profundidad: Las mejores prácticas de SQL

Estas mejores prácticas incluyen directrices para introducir consultas SQL, desarrollar documentación y ejemplos que demuestran estas prácticas. Este es un gran recurso para tener a mano cuando se está utilizando SQL a ti mismo; sólo tiene que ir directamente a la sección correspondiente para revisar estas prácticas. Es como una guía de campo de SQL 

## Mayúsculas y minúsculas

En SQL, las mayúsculas no suelen ser importantes. Puedes escribir **SELECT** o **SELECT** o SeLeCT. ¡Todos funcionan! Pero si utiliza las mayúsculas como parte de un estilo coherente, sus consultas tendrán un aspecto más profesional.

Para introducir consultas SQL como un profesional, siempre es una buena idea utilizar mayúsculas para los inicios de cláusula (por ejemplo, **SELECT**, **FROM**, FROM, etc.). Las Funciones también deben ir en mayúsculas (por ejemplo, **SUM()**). Los nombres de columnas deben ir en minúsculas (consulte la sección sobre **snake_case** más adelante en esta guía). Los nombres de tabla deben ir en **CamelCase** (consulte la sección sobre **CamelCase** más adelante en esta guía). Esto ayuda a que las consultas sean coherentes y más fáciles de leer, al tiempo que no afecta a los datos que se obtendrán al ejecutarlas. La única vez que las mayúsculas importan es cuando están entre comillas (más información sobre las comillas más adelante).

Proveedores de Bases de datos SQL pueden usar variaciones ligeramente diferentes de SQL. Estas variaciones se denominan **dialectos SQL**. Algunos dialectos de SQL distinguen entre mayúsculas y minúsculas. BigQuery es uno de ellos. **Vertica** es otro. Pero la mayoría, como **MySQL**, **PostgreSQL** y **SQL Server**, no distinguen entre mayúsculas y minúsculas. Esto significa que si se busca country_code = ‘us’, devolverá todas las entradas que tengan 'us', 'uS', 'Us', y 'US'. Este no es el caso de BigQuery. BigQuery distingue entre mayúsculas y minúsculas, por lo que esa misma búsqueda sólo devolvería entradas en las que country_code fuera exactamente 'us'. Si country_code es 'US', BigQuery no devolvería esas entradas como parte del resultado.

## Comillas simples o dobles: '' o " "

En la mayoría de los casos, tampoco importa si utiliza comillas simples **' '** o dobles **" "** al referirse a cadenas. Por ejemplo, **SELECT** es un inicio de cláusula. Si pones **SELECT** entre comillas como '**SELECT**' o "**SELECT**", entonces SQL lo tratará como una cadena de texto. Su consulta devolverá un error porque su consulta necesita una cláusula **SELECT**.

Pero hay dos situaciones en las que sí importa el tipo de comillas que utilice:

1. Cuando desee que las cadenas sean identificables en cualquier dialecto SQL.

2. Cuando su cadena contiene un apóstrofo o comillas

Dentro de cada dialecto SQL existen reglas sobre lo que se acepta y lo que no. Pero una regla general en casi todos los dialectos SQL es utilizar comillas simples para las cadenas. Esto ayuda a evitar muchas confusiones. Así, si queremos hacer referencia al país EE.UU. en una cláusula **WHERE** (por ejemplo, country_code = 'US'), utilizaremos comillas simples alrededor de la cadena 'US'.

La segunda situación es cuando la cadena tiene comillas en su interior. Supongamos que tenemos una columna favorite_food en una tabla llamada FavoriteFoods y la otra columna corresponde a cada friend. 

| friend | favorite_food |
| --- | --- |
| Rachel DeSantos | Pastel de pastor |
| Sujin Lee | Tacos |
| Najil Okoro | Paella española |

Puedes observar que la comida favorita de Rachel contiene un apóstrofo. Si utilizara comillas simples en una cláusula **WHERE** para buscar al amigo que tiene esta comida favorita, tendría el siguiente aspecto:

```sql
SELECT
      friend
FROM
      FavoriteFoods
WHERE
      favorite_food = 'Shepherd's pie'
```

Esto no funcionará. Si ejecuta esta consulta, obtendrá un error a cambio. Esto se debe a que SQL reconoce una cadena de texto como algo que comienza con una comilla ' y termina con otra comilla '. Así que en la mala consulta anterior, SQL piensa que el favorite_food que está buscando es 'Shepherd', porque el apóstrofo en Shepherd's termina la cadena.

En general, esta debería ser la única ocasión en la que se utilizarían comillas dobles en lugar de simples. Por lo tanto, su consulta tendría este aspecto:

```sql
SELECT
      friend
FROM
      FavoriteFoods
WHERE
      favorite_food = "Shepherd's pie"
```

SQL entiende que las cadenas de texto pueden empezar con comillas simples ' o con comillas dobles ". Dado que esta cadena comienza con comillas dobles, SQL esperará otra comilla doble para señalar el final de la cadena. Esto mantiene a salvo el apóstrofo, por lo que devolverá "Shepherd's pie" y no 'Shepherd'.

## Comentarios a modo de recordatorio

A medida que te sientas más cómodo con SQL, serás capaz de leer y entender las consultas de un vistazo. Pero nunca está de más tener comentarios en la consulta para recordar lo que estás intentando hacer. Y si compartes tu consulta, también ayuda a otros a entenderla.

Por ejemplo:

```sql
--This is an important query used later to join with the accounts table
SELECT
      rowkey, -key used to join with account_id
      Info.date, -date is in spring format YYYY-MM-DD HH:MM:SS
      Info.code -e.g. 'pub-###'
FROM
      Publishers
```

Puede utilizar **#** en lugar de los dos guiones, **--**, en la consulta anterior, pero tenga en cuenta que **#** no se reconoce en todos los dialectos SQL (MySQL no reconoce #). Así que es mejor utilizar **--** y ser coherente con él. Cuando añades un comentario a una consulta usando **--**, el motor de consulta de la base de datos ignorará todo lo que haya en la misma línea después de **--**. Continuará procesando la consulta a partir de la línea siguiente.

## nombres de columnas en mayúsculas y minúsculas

Es importante asegurarse siempre de que la salida de la consulta tenga nombres fáciles de entender. Si crea una nueva columna (por ejemplo, a partir de un cálculo o de la concatenación de nuevos campos), la nueva columna recibirá un nombre genérico por defecto (por ejemplo, f0). Por ejemplo:

```sql
SELECT
      SUM(tickets),
      COUNT(tickets),
      SUM(tickets) AS total_tickets,
      COUNT(tickets) AS number_of_purchases
FROM
      Purchases
```
Los resultados son:

| f0 | f1 | total_tickets | número_de_compras |
| --- | --- | --- | --- |
| 8 | 4 | 8 | 4 |

Las dos primeras columnas se denominan **f0** y **f1** porque no se nombraron en la consulta anterior. SQL utiliza por defecto f0, f1, f2, f3, y así sucesivamente. Hemos nombrado las dos últimas columnas total_tickets y number_of_purchases para que aparezcan en los resultados de la consulta. Esta es la razón por la que siempre es bueno dar a las columnas nombres útiles, especialmente cuando se utilizan funciones. Después de ejecutar la consulta, querrá poder entender rápidamente los resultados, como las dos últimas columnas que hemos descrito en el ejemplo.

Además, es posible que observe que los nombres de las columnas tienen un guión bajo entre las palabras. Los nombres nunca deben tener espacios. Si total_tickets tuviera un espacio y pareciera total tickets, SQL arrojaría un error de sintaxis porque no sabría qué hacer con la segunda palabra (tickets). Por lo tanto, los espacios son malos en los nombres SQL. Nunca use espacios.

La mejor práctica es utilizar **snake_case**. Esto significa que 'total tickets', que tiene un espacio entre las dos palabras, debe introducirse como total_tickets con un guión bajo en lugar de un espacio.

## Nombres CamelCase para tablas

También puede utilizar las mayúsculas y minúsculas de **CamelCase** para nombrar sus tablas. Las mayúsculas de CamelCase significan que escribes en mayúscula el principio de cada palabra, como un camello de dos jorobas (bactriano). Así que la tabla TicketsByOccasion utiliza mayúsculas CamelCase. Tenga en cuenta que la capitalización de la primera palabra en **CamelCase** es opcional; también se utiliza **camelCase**. Algunas personas diferencian entre los dos estilos llamando a **CamelCase**, **PascalCase**, y reservando **camelCase** para cuando la primera palabra no está en mayúsculas, como un camello de una joroba (dromedario); por ejemplo, ticketsByOccasion.

Al fin y al cabo, **CamelCase** es una elección de estilo. Hay otras formas de nombrar las tablas, como:

* Todo en minúsculas o mayúsculas, como ticketsbyoccasion o TICKETSBYOCCASION

* Con snake_case, como tickets_by_occasion

Ten en cuenta que la opción con todas las letras en minúsculas o mayúsculas puede dificultar la lectura del nombre de tu tabla, por lo que no se recomienda para uso profesional.

La segunda opción, **snake_case**, está técnicamente bien. Con las palabras separadas por guiones bajos, el nombre de la mesa es fácil de leer, pero puede hacerse muy largo porque se van añadiendo guiones bajos. También lleva más tiempo introducirlo. Si utilizas mucho esta tabla, puede convertirse en una tarea pesada.

En resumen, depende de ti utilizar **snake_case** o **CamelCase** al crear nombres de tabla. Sólo asegúrese de que el nombre de su tabla sea fácil de leer y coherente. También asegúrese de averiguar si su empresa tiene una forma preferida de nombrar sus tablas. Si es así, siga siempre su convención de nomenclatura para mantener la coherencia.

## Indentación

Por regla general, la longitud de cada línea de una consulta debe ser <= 100 caracteres. Por ejemplo, observe esta consulta con una línea de más de 100 caracteres:

```sql
SELECT
CASE WHEN genre = 'horror' THEN 'Will not watch' WHEN genre = 'documentary' 
THEN 'Will watch alone' ELSE 'Watch with others' END AS 
watch_category, COUNT(movie_title) AS number_of_movies
    FROM
        MovieTheater
    GROUP BY
        1
```
Esta consulta es difícil de leer e igual de difícil de solucionar o editar. A continuación, se muestra una consulta en la que nos ceñimos a la regla de <= 100 caracteres:

```sql
SELECT
    CASE
        WHEN genre = 'horror' THEN 'Will not watch' 
        WHEN genre = 'documentary' THEN 'Will watch alone' 
        ELSE 'Watch with others' 
        END AS watch_category, COUNT(movie_title) AS number_of_movies
FROM
    MovieTheater
GROUP BY
    1
```

Ahora es mucho más fácil entender lo que se intenta hacer en la cláusula **SELECT**. Por supuesto, ambas consultas se ejecutarán sin problemas porque la indentación no importa en SQL. Pero una indentación adecuada sigue siendo importante para mantener las líneas cortas. Y será valorada por cualquiera que lea tu consulta, ¡incluido tú mismo!

## Comentarios de varias líneas

Si haces comentarios que ocupan varias líneas, puedes utilizar -- para cada línea. O, si tiene más de dos líneas de comentarios, puede que sea más limpio y sencillo utilizar /* para iniciar el comentario y */ para cerrarlo. Por ejemplo, puede utilizar el método -- como se indica a continuación:

```sql
-- Date: September 15, 2020 
-- Analyst: Jazmin Cisneros 
-- Goal: Count the number of rows in the table 
SELECT
	COUNT(*) number of rows -- the * stands for all so count all 
FROM
 	table
```

O bien, puede utilizar el método /* */ como a continuación:

```sql
/* Date: September 15, 2020 
Analyst: Jazmin Cisneros 
Goal: Count the number of rows in the table 
*/
SELECT
	COUNT(*) number of rows -- the * stands for all so count all 
FROM
 	table
```

En SQL, no importa qué método utilices. SQL ignora los comentarios independientemente del método que utilice: #, --, o /* y */. Así que depende de ti y de tus preferencias personales. Los métodos /* y */ para comentarios multilínea suelen tener un aspecto más limpio y ayudan a separar los comentarios de la consulta. Pero no hay un Método correcto o incorrecto.

## Editores de texto SQL

Cuando te unes a una empresa, puedes esperar que cada empresa utilice su propia plataforma SQL y dialecto SQL. La plataforma SQL que utilicen (por ejemplo, BigQuery, MySQL o SQL Server) es donde introducirás y ejecutarás tus consultas SQL. Pero ten en cuenta que no todas las plataformas SQL proporcionan editores de secuencias de comandos nativos para introducir código SQL. Los editores de texto SQL te ofrecen una interfaz donde puedes introducir tus consultas SQL de una forma más sencilla y codificada por colores. De hecho, ¡todo el código con el que hemos estado trabajando hasta ahora fue introducido con un editor de texto SQL!
