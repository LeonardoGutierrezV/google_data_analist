# Más información sobre integridad y conformidad de los datos

Esta lectura ilustra la importancia de la integridad de los datos mediante un ejemplo de los datos de una empresa global. Al final se proporcionarán definiciones de términos relevantes para la integridad de los datos.

## Escenario: fechas de calendario para una empresa global

Las fechas del calendario se representan de muchas formas abreviadas diferentes. Dependiendo de dónde viva, puede que se utilice un formato diferente.

* En algunos países, 12/10/20 (DD/MM/AA) significa 12 de octubre de 2020.

* En otros países, el estándar nacional es AAAA-MM-DD, por lo que 12 de octubre de 2020 se convierte en 2020-10-12.

* En Estados Unidos, (MM/DD/AA) es el formato aceptado por lo que el 12 de octubre de 2020 será 10/12/20.

Ahora, piense qué pasaría si trabajara como analista de datos para una empresa global y no comprobara los formatos de fecha. Pues que la integridad de sus datos sería probablemente cuestionable. Cualquier análisis de los Datos sería inexacto. ¡Imagínese pedir inventario extra para diciembre cuando en realidad se necesitaba en octubre!

Un buen análisis depende de la integridad de los datos, y la integridad de los datos suele depender del uso de un formato común. Por eso es importante comprobar dos veces el formato de las fechas para asegurarse de que lo que usted cree que es el 10 de diciembre de 2020 no es en realidad el 12 de octubre de 2020, y viceversa.

He aquí otras cosas de las que hay que cuidarse:

* **Replicación de datos** que comprometa la integridad de los mismos: Siguiendo con el ejemplo, imagine que pide a sus homólogos internacionales que verifiquen las fechas y se ciñan a un formato. Un analista copia un gran conjunto de datos para comprobar las fechas. Pero por problemas de memoria, en realidad sólo se copia una parte del conjunto de datos. El analista estaría verificando y normalizando datos incompletos. Ese conjunto de datos parcial se certificaría como conforme, pero el conjunto de datos completo seguiría conteniendo fechas que no se han verificado. Dos versiones de un Conjunto de datos pueden introducir resultados incoherentes. Una auditoría final de los resultados sería esencial para revelar lo sucedido y corregir todas las fechas. 

* **Transferencia de datos** que compromete la integridad de los datos: Otro analista comprueba las fechas en una hoja de cálculo y decide importar los datos validados y normalizados de nuevo a la base de datos. Pero supongamos que el campo de fecha de la hoja de cálculo se clasificó incorrectamente como campo de texto durante el proceso de importación (transferencia) de datos. Ahora algunas de las fechas de la base de datos se almacenan como cadenas de texto. En este punto, es necesario limpiar los datos para restaurar su integridad.

* **Manipulación de datos** que compromete la integridad de los mismos: Al comprobar las fechas, otro analista observa lo que parece ser un registro duplicado en la base de datos y lo elimina. Pero resulta que el analista eliminó un registro Único para la filial de una empresa y no un registro duplicado para la empresa. A su Conjunto de datos le faltan ahora datos y hay que restaurar los datos para que estén completos.

## Conclusión

Afortunadamente, con un formato de fecha estándar y el cumplimiento por parte de todas las personas y sistemas que trabajan con los datos, se puede mantener la integridad de los datos. Pero no importa de dónde procedan sus datos, asegúrese siempre de comprobar que son **válidos**, **completos** y **limpios** antes de iniciar cualquier análisis.

## Referencia: Limitaciones de datos y ejemplos

A medida que avance en su viaje de datos, se encontrará con muchos tipos de limitaciones de datos (o criterios que determinan la validez). La tabla siguiente ofrece definiciones y ejemplos de términos de limitaciones de datos con los que podría encontrarse. 

| Limitaciones de datos | Definición | Ejemplos |
| --- | --- | --- |
| Tipo de datos | Los Valores deben ser de un tipo determinado: fecha, número, porcentaje, booleana, etc. | Si el Tipo de datos es una fecha, un solo número como 30 fallaría la restricción y sería inválido |
| Rango de datos | Los Valores deben estar comprendidos entre los valores máximo y mínimo predefinidos | Si el Rango de datos es 10-20, un valor de 30 fallaría la restricción y sería inválido | Obligatorio | Los Valores no pueden dejarse en blanco o vacíos | Si la edad es obligatoria, ese valor debe rellenarse |
| Único | Los Valores no pueden tener un duplicado | Dos personas no pueden tener el mismo número de teléfono móvil dentro de la misma área de servicio |
| Patrones de expresión regular (regex) | Los Valores deben coincidir con un Patrón prescrito | Un número de teléfono debe coincidir con ###-##-#### (no se permiten otros caracteres) |
| Validación de campos cruzados | Deben cumplirse ciertas condiciones para múltiples campos | Los Valores son porcentajes y los valores de varios campos deben sumar 100% |
| Clave primaria | (Sólo bases de datos) el valor debe ser único por columna | Una tabla de base de datos no puede tener dos filas con el mismo valor de clave primaria. Una clave primaria es un identificador en una base de datos que hace referencia a una columna en la que cada valor es único. Más adelante en el programa encontrará más información sobre claves primarias y foráneas. |
| Establecer-membresía | (Sólo bases de datos) los valores para una columna deben proceder de un conjunto de valores discretos | Valores para una columna deben ser Sí, No o No aplicable |
| Clave foránea | (Sólo bases de datos) los valores para una columna deben ser valores únicos procedentes de una columna de otra tabla | En una base de datos de contribuyentes de EE.UU., la columna Estado debe ser un estado o territorio válido con el conjunto de valores aceptables definido en una tabla de Estados independiente| 
| Exactitud | El grado en que los datos se ajustan a la entidad real que se está midiendo o describiendo | Si los valores de los códigos postales se validan por la ubicación de la calle, la exactitud de los datos aumenta. |
| Exhaustividad | El grado en que los datos contienen todos los componentes o medidas deseados | Si los datos para los perfiles personales requieren el color del pelo y de los ojos, y se recogen ambos, los datos están completos. |
| Coherencia | El grado en que los datos son repetibles desde diferentes puntos de entrada o recogida | Si un cliente tiene la misma dirección en las bases de datos de ventas y reparaciones, los datos son coherentes. |
